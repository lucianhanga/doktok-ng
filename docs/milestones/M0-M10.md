# DokTok NG — Milestone Roadmap (M0-M11)

## Rule

Every milestone must produce a functional, runnable version. Do not merge code that leaves the system
unusable. Do not implement multiple milestones in one pass. Do not claim success without running the
available checks.

---

## M0 — Skeleton

Functional result: app starts locally; backend health endpoint works; UI shell loads; PostgreSQL +
pgvector runs through Docker Compose.

Scope: repo skeleton; Python (`uv`) workspace; frontend (`pnpm`) workspace; Docker Compose; config
loading; health endpoint; minimal UI; placeholder interfaces; test/lint/typecheck commands; README
quickstart.

Out of scope: real ingestion, extraction, RAG, MCP server.

Acceptance:

- `docker compose up` starts Postgres + pgvector
- backend responds to `GET /health`
- UI starts and shows a DokTok NG shell
- tests/lint/typecheck commands exist and pass

---

## M1 — Folder Ingestion

Functional result: user drops a file into `storage/files/ingest`; worker detects it and creates an
ingestion job.

Scope: folder watcher; stable-file detection; atomic move to `in.process`; SHA-256 hash; MIME
detection by content; job table; failed/quarantine handling; document/job status API.

Acceptance:

- file is detected and a job is created
- MIME type is recorded
- duplicate hash is handled
- unsupported file goes to failed/quarantine

---

## M2 — Basic Text and PDF Extraction

Functional result: text, Markdown, and born-digital PDFs become active documents.

Scope: direct text parser; direct Markdown parser; PDF embedded-text extraction; canonical artifacts;
document table; document detail API; simple document list UI.

Acceptance:

- `.txt` becomes a searchable text artifact
- `.md` preserves Markdown content
- born-digital PDF text is extracted
- document moves to `docs.active`

---

## M3 — OCR Extraction

Functional result: scanned PDFs and images can be ingested.

Scope: PDF classifier; scanned-PDF OCR; image OCR; mixed-PDF strategy; OCR confidence storage;
normalized/searchable PDF artifact where applicable.

Acceptance:

- scanned PDF produces text
- image produces text
- mixed PDFs do not blindly destroy good embedded text
- failed OCR is visible in failed jobs

---

## M4 — Vector and Full-Text Search

Functional result: user can search documents semantically and by exact terms.

Scope: chunking; Ollama embedding adapter (`qwen3-embedding:0.6b`); pgvector storage; PostgreSQL
full-text indexing; hybrid search API; search UI.

Acceptance:

- semantic query returns relevant chunks
- exact keyword query returns matching chunks
- result includes document and page metadata where available

---

## M5 — Entity Indexing

Functional result: user can search and filter by entities.

Scope: entity extraction adapter (spaCy or rule-based first); people/org/location/date/document-id
extraction; entity table; entity filters; entity search UI.

Acceptance:

- entities are extracted from ingested documents
- entity filters work
- entity search returns linked documents/chunks

---

## M6 — Chat with Documents

Functional result: user can ask questions against documents and get cited answers.

Scope: RAG orchestration; Ollama chat adapter (default `qwen3.6:35b-a3b`); document filters; citation
formatter; chat UI.

Acceptance:

- answer includes citations
- citations link to source chunks/documents
- system refuses unsupported claims when retrieval is insufficient

---

## M7 — Ingestion Dashboard and Hardening

Functional result: user can monitor and manage ingestion.

Scope: ingestion dashboard; retry failed jobs; reindex document; delete document; audit events; better
logs; metrics endpoint; admin controls.

Acceptance:

- failed jobs are visible; retry works; reindex works; audit events are recorded

In progress. Shipped so far: the Overview dashboard now splits the document library from an Ingestion
pipeline view (actionable states only); `DELETE /api/v1/documents/{id}` and
`POST /api/v1/documents/{id}/reingest` (any status) with FK-cascade purge; the Documents tab gained
List + Thumbnails views with multi-select and bulk reingest/delete; per-document thumbnails; keyset-
paginated list with multi-sort + token filtering + select-all-matching; and a Settings tab for
runtime AI model selection (Ollama / opt-in OpenAI). Metrics endpoint and broader admin controls
remain. See the architecture doc (§16–§17), ADR-0013, and ADR-0014.

### M8.x — Pipeline cost and formats (shipped)

Cross-cutting ingestion improvements, runnable on top of M7:

- **First-pages enrichment (#311):** the cheap LLM features (`doc_metadata`, `doc_classify`) read only
  the opening pages of a document, cutting tokens/latency/cost; heavier features still read full text.
  See architecture §16.
- **Entity cleanup (#312):** the regex extractor drops the low-value types (MONEY/DATE/INVOICE_ID/
  CONTRACT_ID/DOCUMENT_ID), keeping only EMAIL/URL; migration `0030` removes existing noise rows. See
  architecture §11.
- **Office-document support (#313, ADR-0019):** `.docx`/`.xlsx`/`.pptx` are converted to PDF on ingest
  by a local Gotenberg container and reuse the canonical PDF path. See architecture §19.

Known follow-up: `structured_records` truncates very long documents to the first 16k chars (#314).

- MCP access is audited

## M11 — Deployment

Functional result: DokTok NG can be deployed to small hardware (a TRIGKEY N95: 4 cores, 8 GB, no GPU)
for a staging and a limited-production environment.

The founding decision is the **hybrid deployment topology** (ADR-0020): OCR (PaddleOCR) and embeddings
(Ollama `qwen3-embedding:0.6b`) stay local on the box, while the enrichment pipeline and RAG chat +
rerank run on OpenAI — because the local LLMs do not fit on 8 GB. The operator guide is
[docs/operations/deployment-trigkey-n95.md](../operations/deployment-trigkey-n95.md).

Scope (the M11 epic owns the ticket list; not duplicated here): production Docker Compose and
backend/worker Dockerfiles; a TLS reverse proxy (Caddy) that injects the bearer token; headless
seeding of the AI settings + OpenAI key (APP-2); reconciling `DOKTOK_NO_EGRESS` so it actually gates
OpenAI egress (APP-3); encrypting the OpenAI key at rest (APP-8); and deploy CI / image publishing.

Buildable today (no new code): the per-purpose provider split (ADR-0014) set via the Settings UI, with
OCR and embeddings hardwired local. Everything under "Scope" above is planned M11 work, not shipped.

Acceptance:

- the app runs on an N95-class box in the hybrid configuration
- content egress to OpenAI is documented and the privacy trade-off is explicit (ADR-0020)
- backups cover the Postgres volume and the file tree; the secret-bearing backup is called out
